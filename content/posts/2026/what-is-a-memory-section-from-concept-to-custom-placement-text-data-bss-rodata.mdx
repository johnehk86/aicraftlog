This time, we're going one level deeper — **how memory is divided into sections**, and the fact that you can actually create your own.

* * *

## What Is a Section?

Look up "section" in the dictionary and you get: **a part of something.**

That's exactly what it is. A memory section is a **designated region of memory for a specific purpose.**

Think of it like a house. A house has a living room, a kitchen, a bedroom, and a storage room. Each room has a different purpose. Memory works the same way — there's a region for code, a region for initialized variables, a region for uninitialized variables. Each one has its own role.

* * *

## Let's See It in Code

Take this simple example:

```c
int Test1;
int Test2 = 2;
const int Test3 = 3;

int main(void)
{
    int Test4 = 4;
    Test1++;
    Test2++;
    Test4++;

    return 0;
}
```

Four variables. Four different sections.

## Why Bother Splitting?

_"Can't we just dump everything in one place?"_

No. Because **each type of data has a different purpose** — and different runtime behavior.

### .bss — Uninitialized globals

`Test1` has no initial value. There's nothing to store in Flash (ROM). The start code just needs to reserve space in RAM and zero it out. No Flash space consumed.

### .data — Initialized globals

`Test2` starts with the value `2`. That value has to live somewhere even before the program runs — so it's stored in Flash. When the program boots, the start code copies it from Flash into RAM. That's why initialized globals need Flash _and_ RAM.

### .rodata — const variables

`Test3` is `const` — read-only. It never needs to be written. It can stay in Flash permanently. The start code doesn't need to allocate any RAM for it.

### Stack — local variables

`Test4` lives inside a function. It's pushed onto the Stack when the function is called and disappears when the function returns.

* * *

## What's the Benefit of Splitting?

Splitting sections enables **efficient use of extremely limited resources.**

*   Variables that don't need initialization → **no wasted Flash**
    
*   Read-only data stays in Flash → **RAM saved**
    
*   Only what's needed gets copied → **faster boot time**
    

In embedded development, both Flash and RAM are scarce. Managing them well is one of the core skills of firmware development.

The `z` prefix and `a0`, `a1`, `a8`, `a9` suffixes? Those are **MCU-specific sections** designed for faster memory access and more granular control inside the chip's memory architecture. Think of them as optimized variants of the standard sections.

## You Can Create Your Own Section

Here's where things get interesting.

> **You can define a custom section and place it at any memory address you want.**

In the TASKING compiler, `#pragma` is how you do it:

```c
#pragma protect on
#pragma section code "test"   // Start a code section named "test"

void TestFunction(void)
{
    cnt++;
}

#pragma protect restore
#pragma section code restore  // End section, restore default
```

After this, `TestFunction` will be placed into a section called `.text.test`.

Compile and check the map file — `.text.test` is right there.

* * *

## Placing a Custom Section at a Specific Address

Creating the section is just step one. The **linker script** controls where it actually lands in memory.

Say you want `TestFunction` at address `0x80001000`. Add this to the linker script:

```
group (ordered, run_addr=0x80001000)
{
    select "*.test";
}
```

Recompile, check the map file again — `.text.test` is now sitting at `0x80001000`.

This works for variables too, not just functions.

* * *

## The Full Picture

Memory sections follow a simple three-step flow:

1.  **The compiler** classifies variables and functions into sections based on their type
    
2.  **The linker script** maps those sections to actual memory addresses
    
3.  **If needed**, you can define custom sections and place them exactly where you want
    

All of this is orchestrated by the linker script.

Managing memory well — knowing what goes where and why — that's what separates good firmware from great firmware. It's one of the most satisfying parts of the job.
